# Review: MARS — A Memory-Augmented Agentic Terminal Multiplexer that Learns From Your Commands

*EMNLP 2026 System Demonstrations track. First-principles review of `paper/main.tex`.*

## Summary

MARS is a single-binary Rust terminal editor + multiplexer with an embedded LLM agent whose one idea is *local memory*: because the agent lives where the user works, it can retrieve from two corpora a cloud assistant structurally cannot reach — the user's own past commands (M1) and the tool's own docs/registry (M2) — using nothing fancier than BM25 few-shot injection. The central empirical claim is an "unreasonable effectiveness" result: on 24 project-specific shell tasks, retrieving the user's own past corrections lifts a free model (Qwen3-32B) from 38% to 100% accuracy while spending 40% fewer tokens. The paper is unusually well-written and honest, the system is real and installable, and the token-cost finding is genuinely interesting — but the headline accuracy result has a leakage problem that must be controlled, the Axis-B table is half-empty, and the evaluation rests on a single model and a self-admittedly noisy judge.

## Overall recommendation

- **Soundness: 3 / 5** — the system is real and the writing is candid about limits, but the flagship 38→100 number is under-controlled for memory-as-oracle leakage, the judge is documented (in the paper's own appendix) to be unstable, n is small, and one of the two result tables is mostly `TBD`.
- **Excitement / Contribution: 3.5 / 5** — a polished, coherent, genuinely usable artifact with a crisp thesis and one non-obvious empirical nugget (grounding cuts tokens). Novelty of the *mechanism* is modest; novelty of the *packaged system* is real.
- **Leaning: Borderline, leaning ACCEPT for the demo track — conditional on the P0 fixes below.** As-is (with a half-`TBD` table and an unguarded headline), it is a borderline-reject on an unfinished evaluation. The gap between those two states is one day of work, almost all of it in §4 and the appendix, not the system.

The demo-track bar is "is there a real, available, reproducible system with a defensible idea?" — MARS clears that. The risk is entirely that the evaluation as presented undercuts its own strongest claim.

## Strengths

- **One crisp, defensible idea, cleanly argued.** "An agent embedded at the terminal can reach local memory a cloud model structurally cannot" (§1, §2.4–2.5) is a real insight, not a slogan. The two-corpus framing (M1 your commands, M2 the tool itself) is tidy and the architecture figure (Fig. 1) sells it in one glance. The "dashed boundary is the thesis" framing is effective.
- **Tasteful system design with honest trade-offs.** §2.1's "the agent is a fifth way of naming an `Action`" and §2.2's registry-validated directive protocol (a `RUN:` that doesn't resolve is rejected pre-dispatch) are the right grounding story for an agent that acts on a workspace. §3's classifier-free sigil routing and the `mars keyd` "key-never-leaves-home" broker are legitimately nice engineering with clearly stated costs (ANSI-over-socket vs. semantic protocol; text directives vs. native tool-calling).
- **The genuinely surprising finding is the token result, and it is robust.** Output tokens falling 295→79 because grounding suppresses a reasoning model's rambling (§4.2) is more interesting and more defensible than the accuracy headline — it doesn't depend on the judge being perfect, and it reframes "memory" as also a compute optimization. This is the paper's best-kept nugget and it is currently undersold relative to the 38→100 line.
- **Refreshingly candid appendix.** The error analysis quotes verbatim from frozen result files, names its own judge's inconsistencies (a59 vs a22/a25; the same output graded correct in `axis_a.jsonl` and partial in `lift.jsonl`), and flags the 7 empty completions as a token-wall artifact. This is exactly the transparency a reviewer wants, and it materially raises trust in the rest.
- **Credible demo-track artifact story.** Single binary, `cargo install mars-terminal`, MIT, ~9,500 LOC, 2.5-minute video, macOS+Linux, works with any OpenAI-compatible or free-tier provider (§1, Availability). The `--selfcheck` "no mocks, drives the real App" design (§3) is a real reproducibility asset.

## Weaknesses / threats to validity

- **[MOST DAMAGING] The 38→100 headline is under-controlled for memory-as-oracle leakage.** The corrective loop stores `(request, command)` pairs (§2.4) and then, at test time, BM25-retrieves against the query. If the seeded store contains a pair whose request is the same or near-lexically-identical to a test item (e.g. stored `"build the release binary" → "cargo build --release"` and test item `"build the release binary"`), then the "warm" condition is a *lookup*, not learning — the answer is literally in the prompt. Table 3's flips ("build the release binary" → `cargo build --release`) are consistent with exactly this. The paper never specifies the protocol: what is in the store during the warm run, how it was generated, and how disjoint the stored requests are from the test requests. Until that is pinned down, "100%" reads as near-tautological, and the paper's own framing ("we did not expect a trivial method to erase the gap", §1) is a surprise the reader cannot verify. **This is the swing factor for the whole evaluation** and is addressable in a day (see P0-1).
- **The evaluation rests on a single free model and a self-admittedly noisy judge.** Every headline number is Qwen3-32B judged by Llama-3.3-70B. The appendix documents the judge being over-strict on placeholder filenames, under-counting cold ecosystem errors (p05, p18 graded correct though wrong), and *unstable across runs on identical outputs*. Since the reported deltas are what the paper sells, judge noise on the order of "a few points either side" (the paper's own words) is not a footnote — it directly bounds the credibility of "46%", "38%", and "75%". The Claude-oracle re-measurement (already planned) helps but does not substitute for a human-agreement number on a sample.
- **Table 2 (Axis B) ships with 4 of 6 cells as red `TBD`.** A table that is mostly placeholders should not go to reviewers; it reads as an unfinished paper and invites a reject reflex regardless of the prose. The appendix even labels the Axis-B numbers "preliminary / illustrative." Either fill the table with the re-measured oracle-judged numbers or cut the table and fold the one solid number (docs 75% on how-do-I) into prose.
- **Table 1's 46% vs. 38% for "personalized" is unexplained.** The top block reports Personalized base = 46% (n=24); the bottom block reports Personalized cold = 38% (n=24). Same set, two different base numbers, no explanation of why (different judge run? the empty-candidate artifact? a different prompt?). A reviewer will read this as an internal inconsistency. (Noted as mid-reconciliation; must be reconciled and, ideally, explained in one line before submission.)
- **No comparison to any off-the-shelf tool, despite positioning against them.** Related Work (§5) contrasts MARS with Warp, Cursor, NL2Bash — but never compares. A reviewer *will* ask: is local BM25-over-your-history actually better than Warp AI's history-aware suggestions or `gh copilot suggest`? These are hard to benchmark head-to-head in a day (Warp/Copilot are closed), so this is addressable-in-rebuttal by *scoping the claim* — MARS's differentiator is the architecture (embedded, persistent, free-model, key-local), not "our retriever beats theirs" — but the paper must say so explicitly rather than leave the implication hanging.
- **Related work has real omissions that dent the novelty claim.** For a paper about NL→shell + corrective, personalized command memory: **GitHub Copilot CLI** (`gh copilot suggest`) is uncited; **Atuin** (a shell-history database with search/sync — arguably the closest prior art to "retrieval over your own commands") is uncited; **`thefuck`** (the canonical "corrective command" tool) is uncited; Amazon Q / Fig and Codeium are unmentioned. Engaging these honestly *strengthens* the paper by sharpening what is actually new (local + free + persistent + editor-integrated + key-broker), rather than looking like the authors were unaware of them.
- **A privacy tension undercuts the security narrative.** §2.4 says MARS reads `$HISTFILE` and injects retrieved lines into the prompt sent to the model. Shell history routinely contains secrets (`export TOKEN=...`, `mysql -p...`, signed URLs). So while the `keyd` broker admirably keeps the *API key* off remote boxes (§3), the memory mechanism can exfiltrate *other* secrets from history into the LLM provider's context. The paper's whole pitch is local/private; this vector deserves an explicit sentence and a mitigation (redaction/denylist), not silence.
- **The "self-instrumenting log IS the benchmark" framing slightly oversells.** The log supplies prompts/contexts/token counts (telemetry); the *gold labels* are authored a-priori (§4.1). That is fine and good, but "the log is the benchmark" (§3, §1) blurs input-telemetry with the labeled gold set. One clarifying clause avoids a reviewer thinking the labels came from the system's own outputs.
- **A demo paper with no screenshot of the demo.** The only figure is an architecture schematic. For the *demonstrations* track, showing the command bar mid-translation, the confirm gate, and a memory retrieval in action would do more to convey the artifact than another paragraph. (Video exists — good — but the PDF should stand alone.)

## Concrete, prioritized suggestions

### P0 — must-fix before submission

1. **Kill the leakage ambiguity on the headline result (§4.2, Table 3).** State the corrective-memory protocol explicitly and make it leakage-controlled: report a **leave-one-out** or **paraphrased-query** setup where the test request is *not* lexically identical to any stored request (e.g., store "compile the optimized build," query "build the release binary"). If the 100% survives paraphrase, you have converted the biggest weakness into your strongest result — a real generalization claim, not a lookup. If it drops, report the honest number; it will still be a large lift and far more credible. Also state how many pairs are in the store and how they were generated. **This is the single highest-leverage change in the paper.**
2. **Fill or cut Table 2.** Ship the oracle-judged Axis-B numbers (Knowledge / Reconfigure / Overall, both columns) or remove the table and state the one robust result (docs memory 75% on how-do-I) in prose. Do not submit a table that is majority `TBD`.
3. **Reconcile Table 1's 46 vs 38** and add a one-line explanation of why base-personalized and cold-personalized differ (or make them the same number). Any unexplained internal inconsistency in the one quantitative table is disproportionately costly.
4. **Reframe the abstract/intro headline from "100%" to a gap-closing claim, and elevate the token result.** Lead with "closes the personalized gap (38%→[oracle number]) *and* cuts tokens 40%," and foreground the 295→79 output-token mechanism as the surprising, judge-independent finding. The limitations section already walks back "100% ≠ perfection at scale" (§4.5) — make the front matter consistent with that honesty so a reviewer isn't the one to catch it.

### P1 — strongly improves

5. **Add a second model under test.** One additional family (a non-reasoning small model, or a different free-tier model) would convert "characterizes the free-tier regime" into a defensible generality claim. This is cheap given the harness already exists and directly answers the most obvious reviewer objection.
6. **Report judge reliability quantitatively.** On a sample (say 30–40 items), report LLM-judge vs. human agreement (or oracle-Claude vs. Llama agreement). The appendix already *narrates* judge noise; a number turns a liability into demonstrated rigor.
7. **Separate the 7 empty completions from accuracy.** Report base accuracy with token-wall blanks excluded (or as retry-recovered), since mixing rate-limit failures into "comprehension accuracy" confounds the metric and, as the appendix notes, understates the base — which artificially inflates the gap memory appears to close.
8. **Fix related work.** Add and honestly engage Copilot CLI (`gh copilot suggest`), Atuin, and `thefuck`; deepen the Warp characterization (Warp AI *does* use command history). Use them to sharpen — not dilute — the novelty: local + free-model + persistent-daemon + editor-integrated + key-broker is the combination none of them offer.
9. **Add a UI screenshot/figure** of the command-bar translation + confirm gate + a memory hit. Demo track rewards showing the thing.
10. **Add one privacy sentence + an Ethics/Broader-Impact note.** Address secrets-in-`$HISTFILE` leaking into prompts (redaction/denylist), state that the command store is plaintext JSONL under `~/.mars`, and note the example corpora contain synthetic org identifiers (`registry.company.com`, `s3://our-bucket`) not real ones.

### P2 — nice-to-have

11. **Release the frozen gold sets and result files** (`axis_a.jsonl`, `lift.jsonl`, `axis_b.jsonl`) and, crucially for credibility, the seeded memory store used in the warm condition. For a demo paper this is the reproducibility clincher.
12. **Confirm the crate is actually published and installable at review time** — reviewers will try `cargo install mars-terminal`.
13. Implement/mention the recency-and-project retrieval weighting the appendix proposes; even as future work it shows the retrieval story has a next step.
14. Clarify the "log is the benchmark" framing (telemetry vs. authored labels) in one clause.

## The one thing that would most raise the score

Run and report the **leakage-controlled corrective-memory experiment** (P0-1): show that the lift survives when the test request is not a lexical copy of a stored one. That single result converts the paper's most attackable claim into its most compelling one, and turns "we put the answer in the prompt" into "the user's own history *generalizes* to close the personalization gap." Everything else on this list is hygiene by comparison.

## Questions to authors (rebuttal-style)

1. **What exactly is in the memory store during the "warm" / "+memory" condition, and how was it generated?** For each of the 24 personalized test items, was there a stored pair with a lexically near-identical request? What is the BM25 overlap between the top retrieved request and the test request? (This determines whether 100% measures learning or lookup.)
2. **Why is personalized base 46% (Table 1 top) but cold 38% (Table 1 bottom) on the same 24 items?**
3. **On a labeled sample, what is the Llama-3.3-70B judge's agreement with a human (or with the Claude oracle)?** Given the cross-run instability you document, what are the confidence intervals on 46%, 38%, and 75%?
4. **Does a translated command's prompt include lines pulled from `$HISTFILE`, and if so, what prevents a secret in shell history from being sent to the (possibly cloud) model?**
5. **How do you position the accuracy contribution against Warp AI and `gh copilot suggest`, both of which already condition on command history?** Is the claim "better retrieval" or "same idea, but local/free/persistent/embedded"? If the latter, can you state it explicitly?
6. **With the empty-completion (token-wall) items removed or retried, what is the corrected base accuracy, and how much of the 38→100 gap does that alone account for?**
7. **Will the gold sets, result JSONL files, and the seeded memory store be released with the artifact?**
