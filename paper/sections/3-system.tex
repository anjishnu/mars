\section{System: The Substrate for Local Memory}
\label{sec:system}

Local memory is only useful if the agent is reliably present where the memory lives,
across disconnects and on machines the user does not own; the architecture guarantees
that presence.

\paragraph{One binary, one core.}
MARS is a single Rust binary whose application core owns all state and never touches a
TTY: input arrives as source-neutral events, and rendering is a stateless projection
onto \code{ratatui} (a Rust terminal-UI library). The same core runs unmodified as a
standalone terminal, a headless daemon, and a test backend, so the evaluation drives
the real system with no mocks (\S\ref{sec:eval}).

\paragraph{A persistent daemon.}
We implemented persistence as a process split rather than as a save-and-restore
feature: a server runs the entire application headless and emits the editor's own ANSI
byte stream over a Unix socket; a thin client owns the real terminal and pumps bytes.
The benefit for memory is direct: because the core keeps running with no client
attached, the command store and the user's session survive a closed laptop, and an
overnight run can be watched and, on request, summarized for the user's return.

\paragraph{Classifier-free routing.}
The command bar routes on a \emph{sigil} rather than a learned classifier: the first
character of the input names the target namespace. \code{!} runs a shell command,
\code{?} asks the agent, \code{@} opens the file navigator, and plain text
fuzzy-searches the action registry. Plain text that matches nothing in the registry
falls back to the translator of \S\ref{sec:translate}, whose proposal the user
confirms before it runs. The fallback stays deterministic, triggered by a registry
miss rather than a classifier score, so there is no routing model to train or explain.

\paragraph{The key never leaves home.}
\emph{A secret is a fact about the user, not the machine, and should live in one place.}
The implementation is a broker. A home-side daemon (\code{mars keyd})
holds the key; when the user runs \code{mars ssh}, MARS reverse-forwards the broker's
socket over the SSH connection, and the agent on the remote box makes its model calls by
proxying each request home, where the broker injects the credential and streams the
completion back. The key never lands on the remote machine at all, so a compromised
box has nothing to exfiltrate; the remaining exposure is the open tunnel, which a
compromised box can use but not steal from, and which closes with the connection.

\paragraph{The log is the benchmark.}
Running with \code{-{}-llm-debug} appends one JSON line per model call (task, model,
token counts, latency, full prompt and reply) to a local stream, and records
whether each suggestion was accepted, edited, or rejected. The log is both a cost
profile the user can inspect and, as \S\ref{sec:eval} shows, the source of every prompt,
context, and token count the evaluation reports: the system instruments itself.

\paragraph{Demonstration.}
The demo walks attendees through the loop the paper measures, live: install MARS with
one \code{cargo} command; reject a wrong translation, correct it, then issue a
paraphrase of the same request and watch the corrected command return from memory; ask
``how do I split the screen?'' and retune the accent color in plain English, both
answered from retrieved documentation; detach and reattach a running session; and
open \code{mars ssh} with the key staying on the presenter's laptop, the debug log
streaming every call in a side pane. The screencast follows the same script.
