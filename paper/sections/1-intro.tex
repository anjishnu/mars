\section{Introduction}

LLM assistants have become a routine part of terminal work. A typical user, an ML
researcher babysitting training runs across a laptop and a handful of remote GPU
machines, asks an assistant to explain a CUDA out-of-memory trace or draft an
\code{rsync} command, and the assistant is the weakest link, for two reasons that
compound. It is generic: it runs in a separate window and sees only the fragment the
user pastes into it. It is also amnesiac: it forgets the command the user corrected
yesterday, forgets ``where was I'' on reattach, and knows nothing about the editor it
sits beside. In the remote setting that defines GPU work there is a third problem:
reaching the work at all usually means exporting an API key onto a machine the user
does not own, where it lingers in the environment and the shell history.

Our observation is that these failures are information gaps, not capability gaps. A
project-specific command is unknowable from the request text: no model, at any scale,
can know that this repository launches training with a particular \code{sbatch}
invocation settled on last week. The missing information exists in exactly one place,
the user's own terminal, in two corpora a browser-tab assistant never keeps: the
history of commands the user actually ran, and the tool's own documentation, action
registry, and configuration schema. Neither corpus is large; what they require is an
agent that lives beside them, persists between requests, and captures corrections as
they happen. Per-invocation CLI agents
\citep{anthropic2024claudecode} live in the shell but do not automatically capture the
corrections a user makes (\S\ref{sec:related}). We built MARS to do all
three\footnote{Code, gold sets, and frozen result files:
\url{https://github.com/anjishnu/mars} (MIT). Downloadable package:
\url{https://crates.io/crates/mars-terminal}. Screencast: [URL].}; it hosts the
user's shell in its panes rather than replacing it.

Framing the gap as informational says precisely what closing it requires: retrieving
the one exemplar that carries the missing fact, and nothing more. We therefore hold the
retriever fixed at plain BM25 over a flat file, as a deliberate control: any gap it
closes is attributable to the information, not to retrieval machinery. Under a
leakage-controlled protocol in which the store holds only \emph{paraphrases} of the
user's earlier requests, retrieval lifts \code{claude-haiku-4-5} from 38\% (9/24) to
96\% (23/24) on project-specific commands, one item short of the verbatim-lookup
ceiling: generalization, not lookup. In a domain this constrained, memory does not
need to be sophisticated to be decisive; it needs to be present.

This paper makes four contributions. First, an open-source, single-binary terminal
agent (\S\ref{sec:agent}) whose memory never leaves the machine: a registry-validated
directive protocol, a corrective-memory loop, and a model cascade. Second, two axes of
\emph{local} memory (\S\ref{sec:m1}--\ref{sec:m2}): the user's own commands, which
close the personalization gap, and the tool's own ground truth, which grounds
self-knowledge and reconfiguration. Third, a leakage-controlled protocol for evaluating
agent memory (\S\ref{sec:eval}): the store holds independently generated paraphrases,
the test uses the original requests, and a verbatim condition bounds the lookup
ceiling. Fourth, the architecture that
makes local memory always present (\S\ref{sec:system}): a persistent daemon, and an SSH
broker that keeps the API key off remote machines. MARS is about 9{,}500 lines of
Rust~\citep{kumar2026mars}.

MARS is private by construction: the memory stores are flat local files the user can
read, edit, and erase; retrieval is lexical, so nothing embedded ever leaves the
machine; the only network call is the completion itself, and even that can stay local
with a local model (\S\ref{sec:directives}); on remote machines the key never leaves
home (\S\ref{sec:system}); and every line that reaches a prompt first passes a
redaction filter (Ethics Statement).
